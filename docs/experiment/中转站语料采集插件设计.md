# 中转站语料采集插件设计（Corpus Tap）

> **文档类型**：模块设计 · **非** 实现代码仓库（实现可独立 repo，本文定义契约与拓扑）  
> **目标**：作为 New API 中转站的 **插上即用** 插件——**简单规则 + 全量收集**；按 **New API 用户** 分区落盘，供离线清洗后做领域微调  
> **范围**：仅 **运营商侧** 中转站 EC2；**不** 替代 New API 计费日志，**不** 在用户侧 Runner 采集  
> **关联**：[EC2-中转站原型实验点设计](./EC2-中转站原型实验点设计.md) · [中转站主流技术栈调研](../research/中转站主流技术栈调研.md)

## 1. 需求确认（与产品含义对齐）

| 你的要求 | 插件职责 |
|----------|----------|
| **每个 New API 用户 ≈ 一个领域桶** | 每条记录 **必须** 带 `user_id`（及 `token_id`）；离线按用户导出即「领域原始矿」 |
| **问答对对微调重要** | 插件 **只存** 原始入站/出站 JSON（含流式重组）；**不** 在网关抽 Q&A、不打质量分 |
| **简单规则 + 全量** | 采集层仅：归属、安全最小脱敏、体积上限、可选采集开关、保留期 |
| **清洗后置** | 去重、去 probe、领域标签、SFT 格式化 → **离线 ETL**，不进入插件 |
| **插上即用** | 不改 New API 源码；用户侧 `base_url` 只改 host/port；Compose 增加服务即可 |

**明确不做**：内容审核 NLP、智能抽样、训练集版本管理、模型微调流水线。

---

## 2. 插件定位

### 2.1 是什么

**Corpus Tap** 是坐在 **客户端 ↔ New API** 之间的 **透明反向代理 + 异步落库 Worker**：

- 对客户端：与直连 New API **同 URL 路径**（`/v1/*` 等），兼容三 Agent 主 wire。  
- 对 New API：普通 HTTP 上游，无需安装扩展、无需改表。  
- 对运营：独立 **语料元数据库** + **对象存储**，与 New API MySQL **物理隔离**。

### 2.2 不是什么

| 不是 | 说明 |
|------|------|
| New API fork / 内置 `ext` | 避免 AGPL 衍生与升级耦合；插件独立发布 |
| 替代 `logs` 消费表 | 计费、渠道统计仍用 New API 管理台 |
| 用户侧 Agent 日志 | 本地 bash/MCP 轨迹不在本插件范围（可选另案） |

## 3. 插上即用拓扑

### 3.1 默认模式：前置透明代理（推荐）

```text
  用户侧 Runner / Agent
        │  Authorization: Bearer <平台 Token>
        │  （可选）X-Corpus-Session-Id: <uuid>   ← 不强制，无则插件生成
        ▼
  ┌─────────────────────────────────────┐
  │  Corpus Tap :8443  (TLS 终止可在此)   │  ← 对用户暴露的 base_url
  │    · 同步转发                       │
  │    · 旁路写入 S3 + corpus-db        │
  └──────────────┬──────────────────────┘
                 ▼
  ┌─────────────────────────────────────┐
  │  New API :3000  (仅内网)             │
  │    MySQL / Redis                    │
  └──────────────┬──────────────────────┘
                 ▼
           上游 Channel
```

**插上步骤（3 步）**：

1. `docker compose` 增加 `corpus-tap`、`corpus-db`（PostgreSQL）、`corpus-minio`（或绑定 AWS S3）。  
2. 将交付用户侧的 `base_url` / `anthropic_base_url` 从 `https://newapi:3000` 改为 `https://corpus-tap:8443`（路径不变）。  
3. 配置 `CORPUS_TAP_UPSTREAM=http://new-api:3000` 与存储连接串。

无需改 Channel、Token、model_mapping。

### 3.2 可选模式：镜像流量（已有 Nginx/Caddy）

若已有反代，可增加 `mirror` / `traffic split` 将副本 POST 发往 Corpus Tap **ingest** 端口（只写不转发）。该模式运维复杂，实验期 **不推荐**。

## 4. 模块拆分

```text
experiment/corpus-tap/
├── proxy/          # 同步转发、限流、请求 ID 注入
├── capture/        # 读写在途 body；流式 SSE 缓冲
├── rules/          # 简单规则引擎（见 §5）
├── enrich/         # 从 New API 响应头/只读库补 user_id、token_id（见 §6）
├── store/
│   ├── meta/       # PostgreSQL 元数据
│   └── blob/       # S3/MinIO 全量 body
└── admin/          # 健康检查、只读统计 API（内网）
```

| 模块 | 职责 |
|------|------|
| **proxy** | 保持与 New API 一致的超时；支持 `RELAY_TIMEOUT` 级长连接 |
| **capture** | 入站 body、上游响应 body、`assembled_stream`（流式结束后） |
| **rules** | 采集前过滤/截断/脱敏（仅 §5 列表） |
| **enrich** | 解析平台 Token → `user_id`；合并 New API 返回的 `request_id` |
| **store** | 异步队列写入，**转发成功优先**，落库失败记 `store_error` 不阻断业务 |
| **admin** | `GET /healthz`；`GET /internal/stats?user_id=`（内网 + 管理密钥） |

## 5. 简单规则（采集期唯一逻辑）

规则在 **写库前** 顺序执行；命中「拒绝」则 **仍转发** New API，仅 **不存** 语料（打标 `skipped_reason` 可选写轻量元数据）。

| ID | 规则 | 默认 | 说明 |
|----|------|------|------|
| R1 | **路径** | 采集 | `POST /v1/chat/completions`、`POST /v1/messages`、`POST /v1/responses`；`GET` 等其他 **不采** |
| R2 | **归属** | 必须解析出 `user_id` | 解析失败：不存全文，记 `enrich_failed`（仍转发） |
| R3 | **采集开关** | 全开 | 环境变量 `CORPUS_TAP_DENY_USER_IDS`、`CORPUS_TAP_DENY_TOKEN_IDS` 逗号分隔 |
| R4 | **安全脱敏** | 开启 | 剥离 `Authorization`；JSON 内常见 key 名（`api_key`、`secret` 等）替换为 `[REDACTED]` |
| R5 | **体积上限** | 32 MiB / body | 超限：存截断版 + `truncated=true`，或整包跳过（可配置 `CORPUS_TAP_ON_OVERSIZE=truncate|skip`） |
| R6 | **保留期** | 90 天 | S3 生命周期 + DB `retention_until`；**不** 在插件内做内容清洗 |
| R7 | **流式** | 全量 | 缓冲 SSE 直至 `[DONE]` 或连接关闭，写 `assembled_stream` 对象 |

**不属于规则（禁止在插件实现）**：最短 prompt 过滤、重复检测、领域分类、语言检测、PII 实体识别。

## 6. 归属与 enrich（用户 = 领域桶键）

### 6.1 原则

微调语料按 **`user_id` 分区**；`token_id` 为细粒度子桶。插件 **不** 维护 `domain` 字段（离线表再加）。

### 6.2 解析顺序

1. **同步**：校验 `Authorization: Bearer`，用与 New API 相同的 Token 校验方式之一：  
   - **推荐**：只读连接 New API MySQL 表 `tokens`（插件仅需 `SELECT`）解析 `user_id`、`token_id`；  
   - **或**：HTTP 调 New API 内部校验接口（若版本提供）。  
2. **响应后**：读取 New API 响应头 / body 中的 `request_id`（依版本而定），写入 `http_exchange.newapi_request_id`。  
3. **可选异步**：从 New API `logs` 只读副本补 `prompt_tokens`、`channel_id`（**非阻塞**）。

### 6.3 会话键（可选）

| 来源 | 字段 |
|------|------|
| 客户端头 `X-Corpus-Session-Id` | 原样作 `session_key` |
| 无 | `hash(user_id, token_id, date, first_request_id)` 或单请求 `session_key = request_id` |

会话仅用于离线 **拼多轮**；插件不解析轮次。

## 7. 数据模型（语料库，独立于 New API MySQL）

实现用 **PostgreSQL 15+**；正文 **仅** 在 S3。

### 7.1 表（最小集）

```sql
-- 插件实例级配置快照（便于复现导出）
CREATE TABLE tap_deployment (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  newapi_image TEXT NOT NULL,
  tap_image TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 一次 HTTP 交换 = 一行（全量采集主表）
CREATE TABLE http_exchange (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  deployment_id UUID REFERENCES tap_deployment(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  -- 归属（领域桶）
  user_id INT NOT NULL,
  token_id INT,
  -- 追踪
  tap_request_id VARCHAR(64) NOT NULL,
  newapi_request_id VARCHAR(64),
  upstream_request_id VARCHAR(128),
  session_key TEXT,
  -- 协议
  endpoint TEXT NOT NULL,
  wire TEXT NOT NULL,  -- openai_chat | anthropic_messages | openai_responses
  is_stream BOOLEAN NOT NULL DEFAULT false,
  status_code INT,
  latency_ms INT,
  model_header TEXT,   -- 请求中的 model 字段（若有）
  -- 存储指针（全量）
  client_request_uri TEXT,
  upstream_response_uri TEXT,
  assembled_stream_uri TEXT,
  client_request_sha256 CHAR(64),
  upstream_response_sha256 CHAR(64),
  -- 规则结果
  truncated BOOLEAN NOT NULL DEFAULT false,
  skipped_reason TEXT,  -- NULL = 已存全文
  store_error TEXT
);

CREATE INDEX idx_exchange_user_time ON http_exchange (user_id, created_at);
CREATE INDEX idx_exchange_session ON http_exchange (session_key);
CREATE INDEX idx_exchange_newapi_rid ON http_exchange (newapi_request_id);
```

### 7.2 S3 对象布局

```text
s3://<bucket>/<deployment_id>/user_id=<uid>/dt=YYYY-MM-DD/
  <exchange_id>/client_request.json.gz
  <exchange_id>/upstream_response.json.gz
  <exchange_id>/assembled_stream.txt.gz   # 流式时
```

### 7.3 离线导出契约（插件只读 API 或 DBT）

按 `user_id` 导出 **原始 JSONL**（清洗管道输入）：

```json
{
  "exchange_id": "uuid",
  "user_id": 12,
  "token_id": 3,
  "session_key": "...",
  "endpoint": "/v1/messages",
  "wire": "anthropic_messages",
  "created_at": "2026-06-03T12:00:00Z",
  "uris": {
    "client_request": "s3://...",
    "upstream_response": "s3://...",
    "assembled_stream": "s3://..."
  }
}
```

**问答对、领域标签、训练 chat template** → 全部由离线 Job 生成。

## 8. 对外接口契约（插上即用）

### 8.1 数据面（与用户侧兼容）

| 项 | 值 |
|----|-----|
| 监听 | `0.0.0.0:8443`（TLS 由前置 Caddy 或 Tap 自身） |
| 路径 | 与 New API 一致，**原样转发** |
| 请求头 | 透传；可选 `X-Corpus-Session-Id` |
| 响应 | 与 New API 一致（含 SSE） |

用户侧 `experiment/user-side/sites.json` 仅改 host，例如：

```json
"base_url": "https://<relay-host>/v1",
"anthropic_base_url": "https://<relay-host>"
```

其中 `<relay-host>` 指向 **Corpus Tap**，而非 `:3000`。

### 8.2 控制面（环境变量）

| 变量 | 必填 | 说明 |
|------|------|------|
| `CORPUS_TAP_UPSTREAM` | ✅ | `http://new-api:3000` |
| `CORPUS_TAP_DATABASE_URL` | ✅ | PostgreSQL DSN |
| `CORPUS_TAP_S3_BUCKET` | ✅ | 桶名 |
| `CORPUS_TAP_S3_REGION` | ✅ | 区域 |
| `CORPUS_TAP_NEWAPI_MYSQL_DSN` | 推荐 | 只读 Token → user 解析 |
| `CORPUS_TAP_MAX_BODY_BYTES` | | 默认 `33554432` (32MiB) |
| `CORPUS_TAP_RETENTION_DAYS` | | 默认 `90` |
| `CORPUS_TAP_DENY_USER_IDS` | | 逗号分隔 |
| `CORPUS_TAP_ADMIN_KEY` | | 内部统计 API |

### 8.3 Compose 片段（实验点落地）

```yaml
services:
  new-api:
    image: calciumion/new-api:<pinned-tag>
    # ... 仅内网暴露 3000

  corpus-tap:
    image: <your-registry>/corpus-tap:<tag>
    environment:
      CORPUS_TAP_UPSTREAM: http://new-api:3000
      CORPUS_TAP_DATABASE_URL: postgres://corpus:***@corpus-db:5432/corpus
      CORPUS_TAP_S3_BUCKET: corpus-raw-prototype
      CORPUS_TAP_NEWAPI_MYSQL_DSN: readonly://@mysql:3306/newapi
    ports:
      - "8443:8443"   # 对用户侧 SG 开放
    depends_on: [new-api, corpus-db]

  corpus-db:
    image: postgres:16-alpine
    volumes: [corpus-pg-data:/var/lib/postgresql/data]
```

### 8.4 健康与就绪

| 端点 | 条件 |
|------|------|
| `GET /healthz` | 200 = 进程存活 |
| `GET /readyz` | upstream ping + DB + S3 均可用 |

部署验收：一条 `curl` 经 Tap 打 `GET /v1/models` 不计入语料；一条 `POST /v1/messages` 后能在 S3 看到对应 `user_id` 前缀对象。

## 9. 与 New API 的关系

| 能力 | New API 自带 | Corpus Tap |
|------|--------------|------------|
| 计费 / 配额 | ✅ `logs` | ❌ |
| Token 统计 | ✅ | ❌ |
| 请求全文 | ❌ | ✅ |
| 按 user 导出 | ❌（仅运营查询） | ✅ 原始 JSONL 索引 |
| 升级耦合 | — | 仅依赖 HTTP 与 Token 表只读 |

**故障策略**：Tap 存储失败 **不得** 导致推理失败；队列积压告警，必要时 `CORPUS_TAP_MODE=proxy-only` 旁路只转发。

## 11. 实施阶段（插件视角）

| 阶段 | 内容 | 产出 |
|------|------|------|
| **P0** | Tap 前置 + 单用户 POST 全量落 S3 | 按 `user_id` 能列出对象 |
| **P1** | 流式 `assembled_stream` + enrich `newapi_request_id` | Codex/Claude 长回复可还原 |
| **P2** | 三端点全开通 + 与用户侧 L4 并行 | 真实 Agent 流量入库 |
| **P3** | 离线导出 CLI + 保留期生命周期 | 交付数据工程清洗 |

## 12. 风险与局限

| 风险 | 缓解 |
|------|------|
| 流式内存 / 磁盘 | 超大 SSE 走临时文件；R5 上限 |
| Token 解析与 New API 版本漂移 | 锁定 digest；集成测试 |
| 用户 ≠ 领域 | 离线加 `domain_id`；插件只保证 `user_id` |
| 上游 ToS / 二次训练 | 法务 + 用户授权；与插件无关但须在运营流程 |
| AGPL | 插件独立仓库，不链接 New API 代码 |

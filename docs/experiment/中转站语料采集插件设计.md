# 中转站语料采集插件设计（Corpus Tap）

> **文档类型**：模块设计 · **非** 实现代码仓库（实现可独立 repo，本文定义契约与拓扑）  
> **目标**：作为 New API 中转站的 **插上即用** 插件——**简单规则 + 全量收集**；按 **New API 用户** 分区落盘，供离线清洗后做领域微调；可选 **用户画像** 离线分析（**独立 LLM 出口**，不复用平台 Channel/Token）  
> **范围**：仅 **运营商侧** 中转站 EC2；**不** 替代 New API 计费日志，**不** 在用户侧 Runner 采集；画像分析 **不** 进入 Tap 热路径

## 1. 需求确认（与产品含义对齐）

| 你的要求 | 插件职责 |
|----------|----------|
| **每个 New API 用户 ≈ 一个领域桶** | 每条记录 **必须** 带 `user_id`（及 `token_id`）；离线按用户导出即「领域原始矿」 |
| **问答对对微调重要** | 插件 **只存** 原始入站/出站 JSON（含流式重组）；**不** 在网关抽 Q&A、不打质量分 |
| **简单规则 + 全量** | 采集层仅：归属、安全最小脱敏、体积上限、可选采集开关、保留期 |
| **清洗后置** | 去重、去 probe、领域标签、SFT 格式化 → **离线 ETL**，不进入 Tap 采集层 |
| **插上即用** | 不改 New API 源码；用户侧 `base_url` 只改 host/port；Compose 增加服务即可 |
| **用户画像（可选）** | 基于已落盘语料，按 `user_id` 聚合行为与主题摘要；**专用分析 LLM**（见 §8），**不** 走 New API、**不** 扣用户平台额度 |

**明确不做**：内容审核 NLP、智能抽样、训练集版本管理、模型微调流水线；**在 Tap 同步路径内** 调用任何 LLM；用 **平台 Channel / 用户 Token** 做画像推理。

## 2. 插件定位

### 2.1 是什么

**Corpus Tap** 是坐在 **客户端 ↔ New API** 之间的 **透明反向代理 + 异步落库 Worker**：

- 对客户端：与直连 New API **同 URL 路径**（`/v1/*` 等），兼容三 Agent 主 wire。  
- 对 New API：普通 HTTP 上游，无需安装扩展、无需改表。  
- 对运营：独立 **语料元数据库** + **对象存储**，与 New API MySQL **物理隔离**。

### 2.2 不是什么

| 不是 | 说明 |
|------|------|
| New API fork / 内置 `ext` | 避免 AGPL 衍生与升级耦合；插件独立发布 |
| 替代 `logs` 消费表 | 计费、渠道统计仍用 New API 管理台 |
| 用户侧 Agent 日志 | 本地 bash/MCP 轨迹不在本插件范围（可选另案） |
| 平台推理复用 | 画像 Worker **禁止** 将分析请求发往 `CORPUS_TAP_UPSTREAM` 或用户 Bearer |

### 2.3 用户画像分析（Profile Analyzer）

**Profile Analyzer** 是语料栈上的 **可选离线 Worker**：

- **输入**：只读 `corpus-db` + S3（`http_exchange` 与 body 对象）；**不** 订阅 Tap 内存队列。  
- **输出**：按 `user_id` 的画像快照（JSON）写入语料库扩展表或 `s3://.../profiles/`；供运营看板、领域桶优先级、清洗规则提示。  
- **LLM**：运营商单独配置的 **分析模型端点**（自有 Key、自有 `base_url`），与 New API Channel、用户 `Authorization` **零复用**（§8.2）。  
- **时序**：默认 **异步批处理**（cron / 队列）；**不** 阻塞用户推理，**不** 改变 Tap 转发语义。

画像结论 **不** 回写 New API 用户表；微调仍依赖 §7.3 原始 JSONL 导出。

## 3. 插上即用拓扑

### 3.1 默认模式：前置透明代理（推荐）

```text
  用户侧 Runner / Agent
        │  Authorization: Bearer <平台 Token>
        │  （可选）X-Corpus-Session-Id: <uuid>   ← 不强制，无则插件生成
        ▼
  ┌─────────────────────────────────────┐
  │  Corpus Tap :8443  (TLS 终止可在此)   │  ← 对用户暴露的 base_url
  │    · 同步转发                       │
  │    · 旁路写入 S3 + corpus-db        │
  └──────────────┬──────────────────────┘
                 ▼
  ┌─────────────────────────────────────┐
  │  New API :3000  (仅内网)             │
  │    MySQL / Redis                    │
  └──────────────┬──────────────────────┘
                 ▼
           上游 Channel
```

**插上步骤（3 步）**：

1. `docker compose` 增加 `corpus-tap`、`corpus-db`（PostgreSQL）、`corpus-minio`（或绑定 AWS S3）。  
2. 将交付用户侧的 `base_url` / `anthropic_base_url` 从 `https://newapi:3000` 改为 `https://corpus-tap:8443`（路径不变）。  
3. 配置 `CORPUS_TAP_UPSTREAM=http://new-api:3000` 与存储连接串。

无需改 Channel、Token、model_mapping。

### 3.2 可选模式：镜像流量（已有 Nginx/Caddy）

若已有反代，可增加 `mirror` / `traffic split` 将副本 POST 发往 Corpus Tap **ingest** 端口（只写不转发）。该模式运维复杂，实验期 **不推荐**。

### 3.3 画像分析拓扑（与数据面隔离）

```text
  用户侧 Agent ──► Corpus Tap :8443 ──► New API :3000 ──► 上游 Channel
                        │
                        ▼（异步落库）
                  corpus-db + S3

  ┌─────────────────────────────────────────────────────────┐
  │  corpus-profile（Profile Analyzer，内网，无用户端口）      │
  │    · 只读 corpus-db / S3                                 │
  │    · 抽样 + 脱敏后调用「分析 LLM」（独立 base_url + Key）   │
  │    · 写 user_profile / profile_run                       │
  └──────────────────────────┬──────────────────────────────┘
                             │ HTTPS（G2 单独白名单，见 §11）
                             ▼
                    分析 LLM（Bedrock / 自建 / 第三方运营 Key）
                    ✗ 不经 New API  ✗ 不带用户 Bearer
```

用户侧 `base_url` **仅** 指向 Tap；画像服务 **不对** 用户侧 SG 开放。

## 4. 模块拆分

```text
experiment/corpus-tap/
├── proxy/          # 同步转发、限流、请求 ID 注入
├── capture/        # 读写在途 body；流式 SSE 缓冲
├── rules/          # 简单规则引擎（见 §5）
├── enrich/         # 从 New API 响应头/只读库补 user_id、token_id（见 §6）
├── store/
│   ├── meta/       # PostgreSQL 元数据
│   └── blob/       # S3/MinIO 全量 body
├── admin/          # 健康检查、只读统计 API（内网）
└── profile/        # 用户画像 Worker（离线；独立 LLM 客户端，见 §8）
```

| 模块 | 职责 |
|------|------|
| **proxy** | 保持与 New API 一致的超时；支持 `RELAY_TIMEOUT` 级长连接 |
| **capture** | 入站 body、上游响应 body、`assembled_stream`（流式结束后） |
| **rules** | 采集前过滤/截断/脱敏（仅 §5 列表） |
| **enrich** | 解析平台 Token → `user_id`；合并 New API 返回的 `request_id` |
| **store** | 异步队列写入，**转发成功优先**，落库失败记 `store_error` 不阻断业务 |
| **admin** | `GET /healthz`；`GET /internal/stats?user_id=`（内网 + 管理密钥） |
| **profile** | 按 `user_id` 批处理画像；**仅** 调用 `CORPUS_PROFILE_*` 分析 LLM；**不** import `proxy` 转发链 |

## 5. 简单规则（采集期唯一逻辑）

规则在 **写库前** 顺序执行；命中「拒绝」则 **仍转发** New API，仅 **不存** 语料（打标 `skipped_reason` 可选写轻量元数据）。

| ID | 规则 | 默认 | 说明 |
|----|------|------|------|
| R1 | **路径** | 采集 | `POST /v1/chat/completions`、`POST /v1/messages`、`POST /v1/responses`；`GET` 等其他 **不采** |
| R2 | **归属** | 必须解析出 `user_id` | 解析失败：不存全文，记 `enrich_failed`（仍转发） |
| R3 | **采集开关** | 全开 | 环境变量 `CORPUS_TAP_DENY_USER_IDS`、`CORPUS_TAP_DENY_TOKEN_IDS` 逗号分隔 |
| R4 | **安全脱敏** | 开启 | 剥离 `Authorization`；JSON 内常见 key 名（`api_key`、`secret` 等）替换为 `[REDACTED]` |
| R5 | **体积上限** | 32 MiB / body | 超限：存截断版 + `truncated=true`，或整包跳过（可配置 `CORPUS_TAP_ON_OVERSIZE=truncate|skip`） |
| R6 | **保留期** | 90 天 | S3 生命周期 + DB `retention_until`；**不** 在插件内做内容清洗 |
| R7 | **流式** | 全量 | 缓冲 SSE 直至 `[DONE]` 或连接关闭，写 `assembled_stream` 对象 |

**不属于规则（禁止在插件实现）**：最短 prompt 过滤、重复检测、领域分类、语言检测、PII 实体识别。

## 6. 归属与 enrich（用户 = 领域桶键）

### 6.1 原则

微调语料按 **`user_id` 分区**；`token_id` 为细粒度子桶。插件 **不** 维护 `domain` 字段（离线表再加）。

### 6.2 解析顺序

1. **同步**：校验 `Authorization: Bearer`，用与 New API 相同的 Token 校验方式之一：  
   - **推荐**：只读连接 New API MySQL 表 `tokens`（插件仅需 `SELECT`）解析 `user_id`、`token_id`；  
   - **或**：HTTP 调 New API 内部校验接口（若版本提供）。  
2. **响应后**：读取 New API 响应头 / body 中的 `request_id`（依版本而定），写入 `http_exchange.newapi_request_id`。  
3. **可选异步**：从 New API `logs` 只读副本补 `prompt_tokens`、`channel_id`（**非阻塞**）。

### 6.3 会话键（可选）

| 来源 | 字段 |
|------|------|
| 客户端头 `X-Corpus-Session-Id` | 原样作 `session_key` |
| 无 | `hash(user_id, token_id, date, first_request_id)` 或单请求 `session_key = request_id` |

会话仅用于离线 **拼多轮**；插件不解析轮次。

## 7. 数据模型（语料库，独立于 New API MySQL）

实现用 **PostgreSQL 15+**；正文 **仅** 在 S3。

### 7.1 表（最小集）

```sql
-- 插件实例级配置快照（便于复现导出）
CREATE TABLE tap_deployment (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  newapi_image TEXT NOT NULL,
  tap_image TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 一次 HTTP 交换 = 一行（全量采集主表）
CREATE TABLE http_exchange (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  deployment_id UUID REFERENCES tap_deployment(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  -- 归属（领域桶）
  user_id INT NOT NULL,
  token_id INT,
  -- 追踪
  tap_request_id VARCHAR(64) NOT NULL,
  newapi_request_id VARCHAR(64),
  upstream_request_id VARCHAR(128),
  session_key TEXT,
  -- 协议
  endpoint TEXT NOT NULL,
  wire TEXT NOT NULL,  -- openai_chat | anthropic_messages | openai_responses
  is_stream BOOLEAN NOT NULL DEFAULT false,
  status_code INT,
  latency_ms INT,
  model_header TEXT,   -- 请求中的 model 字段（若有）
  -- 存储指针（全量）
  client_request_uri TEXT,
  upstream_response_uri TEXT,
  assembled_stream_uri TEXT,
  client_request_sha256 CHAR(64),
  upstream_response_sha256 CHAR(64),
  -- 规则结果
  truncated BOOLEAN NOT NULL DEFAULT false,
  skipped_reason TEXT,  -- NULL = 已存全文
  store_error TEXT
);

CREATE INDEX idx_exchange_user_time ON http_exchange (user_id, created_at);
CREATE INDEX idx_exchange_session ON http_exchange (session_key);
CREATE INDEX idx_exchange_newapi_rid ON http_exchange (newapi_request_id);
```

### 7.2 S3 对象布局

```text
s3://<bucket>/<deployment_id>/user_id=<uid>/dt=YYYY-MM-DD/
  <exchange_id>/client_request.json.gz
  <exchange_id>/upstream_response.json.gz
  <exchange_id>/assembled_stream.txt.gz   # 流式时
```

### 7.3 离线导出契约（插件只读 API 或 DBT）

按 `user_id` 导出 **原始 JSONL**（清洗管道输入）：

```json
{
  "exchange_id": "uuid",
  "user_id": 12,
  "token_id": 3,
  "session_key": "...",
  "endpoint": "/v1/messages",
  "wire": "anthropic_messages",
  "created_at": "2026-06-03T12:00:00Z",
  "uris": {
    "client_request": "s3://...",
    "upstream_response": "s3://...",
    "assembled_stream": "s3://..."
  }
}
```

**问答对、领域标签、训练 chat template** → 全部由离线 Job 生成。

### 7.4 画像表（扩展，与 `http_exchange` 同库）

```sql
-- 一次画像批跑（可手动 / cron 触发）
CREATE TABLE profile_run (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  deployment_id UUID REFERENCES tap_deployment(id),
  started_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  finished_at TIMESTAMPTZ,
  trigger TEXT NOT NULL,  -- cron | manual | backfill
  llm_provider TEXT NOT NULL,  -- 如 bedrock | openai_compatible（仅记录类型，非平台 Channel）
  llm_model TEXT NOT NULL,
  exchange_window_start TIMESTAMPTZ,
  exchange_window_end TIMESTAMPTZ,
  users_processed INT NOT NULL DEFAULT 0,
  llm_requests INT NOT NULL DEFAULT 0,
  error_summary TEXT
);

-- 每个 user_id 当前有效画像（新版本插入后可将旧行标 superseded）
CREATE TABLE user_profile (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id INT NOT NULL,
  profile_run_id UUID NOT NULL REFERENCES profile_run(id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  superseded_at TIMESTAMPTZ,
  -- 结构化摘要（由分析 LLM 生成 + 少量确定性统计合并）
  profile_json JSONB NOT NULL,
  profile_json_uri TEXT,  -- 可选：大对象放 S3
  stats_json JSONB NOT NULL,  -- 确定性：请求数、wire 分布、model 频次、活跃时段等
  sample_exchange_ids UUID[]  -- 本次送入 LLM 的 exchange 样本（审计用）
);

CREATE UNIQUE INDEX idx_user_profile_active
  ON user_profile (user_id) WHERE superseded_at IS NULL;
```

`profile_json` 建议字段（由 Prompt 约束输出 schema，非 Tap 采集期写入）：

| 字段 | 说明 |
|------|------|
| `primary_domains` | 推断主领域标签（如 backend / data / mobile） |
| `tech_stack_hints` | 语言、框架、云厂商等（来自 prompt 抽样） |
| `agent_wire_preference` | openai_chat / anthropic_messages / openai_responses 占比叙述 |
| `task_patterns` | 调试 / 重构 / 文档 / 探索 等粗分类 |
| `risk_flags` | 仅运营向：疑似密钥、极端 prompt 长度等（**不** 自动阻断推理） |
| `confidence` | `low` \| `medium` \| `high` |
| `summary_zh` | 一段可读摘要（运营看板） |

## 8. 用户画像分析（Profile Analyzer）

### 8.1 设计原则

| 原则 | 说明 |
|------|------|
| **与平台推理隔离** | 分析请求 **不得** 使用 New API `:3000`、`CORPUS_TAP_UPSTREAM`、任何用户 `Authorization`、任何 Channel 凭据 |
| **与 Tap 热路径隔离** | `corpus-tap` 进程 **不** 链接分析 LLM SDK；画像由 **`corpus-profile`** 独立容器/二进制执行 |
| **只读语料** | Worker 对 `corpus-db`、S3 仅 `SELECT` / `GetObject`；**不** 改 `http_exchange` |
| **可关可开** | 未配置 `CORPUS_PROFILE_LLM_BASE_URL` 时，Compose **可不** 启动 `corpus-profile`，采集栈仍完整 |
| **成本可控** | 默认抽样 + 每用户每周期请求上限（§8.4）；全量逐条送 LLM **禁止** 为默认 |

### 8.2 独立 LLM 接口契约

分析 LLM 使用 **OpenAI Chat Completions 兼容** 或 **Anthropic Messages** 之一（由 `CORPUS_PROFILE_LLM_WIRE` 固定），与对用户暴露的 wire **无关**——仅服务于运营侧批处理。

| 项 | 平台推理（用户经 Tap） | 画像分析 LLM |
|----|------------------------|--------------|
| 入口 | Corpus Tap → New API | `CORPUS_PROFILE_LLM_BASE_URL` 直连 |
| 鉴权 | 用户平台 Token | `CORPUS_PROFILE_LLM_API_KEY`（运营 Secret） |
| 模型 | 用户请求的 `model` | `CORPUS_PROFILE_LLM_MODEL` |
| 计费归属 | New API `logs` / Channel | 运营商分析预算（独立账单） |
| 出站 SG | new-api → 上游 | **corpus-profile** → 分析端点 FQDN（单独 G2 行） |

**环境变量（仅 `corpus-profile`）**：

| 变量 | 必填 | 说明 |
|------|------|------|
| `CORPUS_PROFILE_DATABASE_URL` | ✅ | 与 Tap 同 PostgreSQL（只读角色即可） |
| `CORPUS_PROFILE_S3_*` | ✅ | 读语料对象；写 `profiles/` 前缀可选 |
| `CORPUS_PROFILE_LLM_BASE_URL` | ✅ | 分析 API 根，如 `https://bedrock-runtime...` 或 `https://analysis.internal/v1` |
| `CORPUS_PROFILE_LLM_API_KEY` | ✅ | 运营 Key；**不进** Git |
| `CORPUS_PROFILE_LLM_MODEL` | ✅ | 分析模型 ID |
| `CORPUS_PROFILE_LLM_WIRE` | | `openai_chat`（默认）\| `anthropic_messages` |
| `CORPUS_PROFILE_MAX_LLM_CALLS_PER_USER` | | 默认 `3` / 次 `profile_run` |
| `CORPUS_PROFILE_SAMPLE_EXCHANGES` | | 默认 `20`（从时间窗内按分层抽样） |
| `CORPUS_PROFILE_DENY_USER_IDS` | | 与 Tap 拒绝列表可对齐 |

**禁止的环境变量组合**（启动时校验失败）：

- `CORPUS_PROFILE_LLM_BASE_URL` 指向 `CORPUS_TAP_UPSTREAM` 或 `http://new-api:3000`  
- `CORPUS_PROFILE_LLM_API_KEY` 与 New API 平台 Token / Channel Key **相同**（可选指纹比对，防误配）

### 8.3 处理流水线

```text
1. 选定时间窗 [t0, t1] 与 user_id 列表（或全量活跃用户）
2. 确定性 stats_json：聚合 http_exchange（计数、wire、model、latency 分位）
3. 按 P1–P4 规则从 S3 拉取抽样 body → 二次脱敏（§8.4）
4. 构造「分析 Prompt」：仅含 stats + 抽样片段 + 输出 JSON schema
5. 调用分析 LLM（1..N 次，受 MAX_LLM_CALLS 限制）
6. 校验 profile_json schema → UPSERT user_profile，关闭旧行 superseded_at
7. 记录 profile_run 指标
```

**不** 在流水线中调用 New API 的 `/v1/chat/completions` 代理能力；若分析模型托管在 Bedrock，Worker 使用 **AWS SDK / 独立 VPCE**，与 new-api 进程 **分账、分角色、分 SG**。

### 8.4 画像期规则（区别于 Tap §5）

| ID | 规则 | 默认 | 说明 |
|----|------|------|------|
| P1 | **触发** | 日批 | `cron` 或 `POST /internal/profile/run`（内网 + `CORPUS_TAP_ADMIN_KEY` 或专用 `CORPUS_PROFILE_ADMIN_KEY`） |
| P2 | **抽样** | 分层随机 | 按 `wire`、是否流式、`model_header` 分层；优先近 7 天 |
| P3 | **送 LLM 前脱敏** | 开启 | 在 Tap R4 基础上，截断单条 prompt 至 8k 字符；剥离路径中的邮箱、内网 IP |
| P4 | **LLM 预算** | 每用户 ≤3 次 | 超限只写 `stats_json`，`profile_json.confidence=low` |
| P5 | **失败** | 不阻断采集 | 画像失败 **不影响** Tap / New API；仅 `profile_run.error_summary` |

### 8.5 对内 API（可选）

| 端点 | 说明 |
|------|------|
| `POST /internal/profile/run` | Body：`{ "user_ids": [12], "window_days": 7 }`；异步返回 `profile_run_id` |
| `GET /internal/profile/user/{user_id}` | 返回当前有效 `user_profile` + `stats_json` |
| `GET /healthz` | 进程存活；**不** 依赖分析 LLM 可达（避免 cron 误杀） |

### 9.3 Compose 片段（实验点落地）

```yaml
services:
  new-api:
    image: calciumion/new-api:<pinned-tag>
    # ... 仅内网暴露 3000

  corpus-tap:

      CORPUS_TAP_UPSTREAM: http://new-api:3000

      - "8443:8443"   # 对用户侧 SG 开放

  corpus-profile:   # 可选；画像分析，独立 LLM（§8）
    image: <your-registry>/corpus-tap:<tag>  # 同镜像不同 entrypoint，或 corpus-profile 镜像
    command: ["corpus-profile", "worker"]
    environment:
      CORPUS_PROFILE_DATABASE_URL: postgres://corpus_ro:***@corpus-db:5432/corpus
      CORPUS_PROFILE_S3_BUCKET: corpus-raw-prototype

      CORPUS_PROFILE_LLM_API_KEY: ${CORPUS_PROFILE_LLM_API_KEY}
```

### 9.4 健康与就绪

| 端点 | 条件 |
|------|------|
| `GET /healthz` | 200 = 进程存活 |
| `GET /readyz` | upstream ping + DB + S3 均可用 |

部署验收：一条 `curl` 经 Tap 打 `GET /v1/models` 不计入语料；一条 `POST /v1/messages` 后能在 S3 看到对应 `user_id` 前缀对象。

## 10. 与 New API 的关系

| 能力 | New API 自带 | Corpus Tap | Profile Analyzer |
|------|--------------|------------|------------------|
| 计费 / 配额 | ✅ `logs` | ❌ | ❌（走分析 LLM 独立账单） |
| Token 统计 | ✅ | ❌ | ❌ |
| 请求全文 | ❌ | ✅ | ❌（只读已落盘副本） |
| 按 user 导出 | ❌（仅运营查询） | ✅ 原始 JSONL 索引 | ✅ `user_profile` 摘要 |
| 推理路径 | Channel 上游 | 透传 New API | **不经过** New API |
| 升级耦合 | — | HTTP + Token 表只读 | 仅语料 schema + 分析 LLM 契约 |

**故障策略**：Tap 存储失败 **不得** 导致推理失败；队列积压告警，必要时 `CORPUS_TAP_MODE=proxy-only` 旁路只转发。画像 Worker 失败 **不得** 影响 Tap / New API。

## 11. 出站与安全（并入中转站审计）

在 [EC2-中转站原型实验点设计](./EC2-中转站原型实验点设计.md) G2 基础上增加：

| 源 | 目标 |
|----|------|
| corpus-tap | new-api:3000（内网） |
| corpus-tap | S3 / MinIO（VPC endpoint 或同 AZ） |
| corpus-tap | corpus-db:5432（内网） |
| corpus-tap | new-api MySQL（**只读** 账号，仅 `tokens` 等最小表） |
| corpus-profile | corpus-db:5432（**只读**）、S3 **读** + `profiles/` **写** |
| corpus-profile | 分析 LLM FQDN:443（**独立** 白名单；**不得** 与 new-api 上游列表混用同一 Channel Key） |

密钥不进 Git；S3 桶 **阻断公网 ACL**；管理统计 / 画像 API 不对用户侧 SG 开放；`CORPUS_PROFILE_LLM_API_KEY` 与平台 Token **分 Secret 存储**。

## 12. 实施阶段（插件视角）

| 阶段 | 内容 | 产出 |
|------|------|------|
| **P0** | Tap 前置 + 单用户 POST 全量落 S3 | 按 `user_id` 能列出对象 |
| **P1** | 流式 `assembled_stream` + enrich `newapi_request_id` | Codex/Claude 长回复可还原 |
| **P2** | 三端点全开通 + 与用户侧 L4 并行 | 真实 Agent 流量入库 |
| **P3** | 离线导出 CLI + 保留期生命周期 | 交付数据工程清洗 |
| **P4** | `corpus-profile` + `profile_run` / `user_profile` 表迁移 | 单用户日批画像；验证 **未** 产生 New API `logs` 增量 |

## 13. 风险与局限

| 风险 | 缓解 |
|------|------|
| 流式内存 / 磁盘 | 超大 SSE 走临时文件；R5 上限 |
| Token 解析与 New API 版本漂移 | 锁定 digest；集成测试 |
| 用户 ≠ 领域 | 离线加 `domain_id`；插件只保证 `user_id`；画像 `primary_domains` 仅作提示 |
| 上游 ToS / 二次训练 | 法务 + 用户授权；与插件无关但须在运营流程 |
| AGPL | 插件独立仓库，不链接 New API 代码 |
| 分析 LLM 误配为平台上游 | 启动校验 `CORPUS_PROFILE_LLM_BASE_URL`；分 SG / 分 IAM |
| 画像抽样泄露敏感片段 | P3 二次脱敏；`CORPUS_PROFILE_DENY_USER_IDS`；审计 `sample_exchange_ids` |
